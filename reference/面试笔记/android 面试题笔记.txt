AsyncTask:   android1.8 以前是串行， android 1.8以后改成了并行， 3.0后又改成了串行
             早期 Android（2.x）：AsyncTask 是 默认并发执行
             大量 App 滥用 AsyncTask  结果：  CPU 抢占严重  UI 掉帧 电量暴跌  👉 这正是 Android 3.0 改为“默认串行”的原因
             主线程： execute()   onPreExecute()
             后台线程：doInBackground()
             主线程：onPostExecute()
             AsyncTask中有两个线程池（SerialExecutor（内部没有线程，只是任务调度器）和THREAD_POOL_EXECUTOR）和一个
             Handler（InternalHandler），其中线程池SerialExecutor用于任务的排队，而线程池
             THREAD_POOL_EXECUTOR用于真正地执行任务，InternalHandler用于将执行环境从线程池切换
             到主线程。

 android的四大组件：activity，service，广播接收器，内容提供者。
        这四大组件都必须在manifest中注册，但是动态广播不需要注册。
        静态广播的设计初衷是为了让app在没有启动的情况下，也可以收到系统广播，但是因为开发者滥用，现在静态广播的接收功能，基本上被完全限制了。

android静态广播的限制历史：
      1. 如果一个 App 刚安装还没运行过，或者被用户在“设置”中点击了“强行停止” ，该 App 就会进入Stopped 状态。处于Stopped 状态的 App，其静态 Receiver 根本收不到任何广播
      2. Android 8.0 (API 26) 的隐式广播禁令：除了开机广播 (BOOT_COMPLETED) 等极少数豁免名单外，App 不能再通过静态注册来接收隐式广播（即没有明确指定包名的广播）
      3. 自启动管理限制：如果app没有自启动权限，即使满足所有条件也无法在没有启动的情况下收到静态广播
      4. 后台启动限制：Android 10/12/14持续加强。在广播中启动activity和服务会直接抛异常。

Android 跨进程通信机制：Binder机制。
      APP进程之极都是内存隔离的，如果需要夸进程通信，一般是通过中间进程，向目标进程发送信息。这样会有两次数据拷贝，中间进程一次，目标进程一次。
      Android为了提高跨进程通信的效率，开发了Binder机制进行跨进程通信，通过内核进程和目标进程内存共享，实现了通信过程只拷贝一次数据。

 apk包体积缩减方案从三个方向入手：
 1. 代码：
    1.开启混淆，对代码内容进行压缩，
    2.删除不必要的第三方代码库依赖，如果第三方库比较大，但是使用的功能简单，可以把这部分代码单独抽签到项目。
 2. 资源文件：

    2. 有的老项目使用了多套图片适应不同的分辨率，但是如果不是为了极致性能，只使用一套1080P基础分辨率的就行了
    3. 对图片使用专门的压缩工具进行压缩，在不损伤图片质量的前提下，尽可能减少图片体积，以及转换为webp图片
 3.架构
    1. gradle打包只配置需要的arm架构（"armeabi-v7a", "arm64-v8a"）, x86系列不需要


MVC模式：把页面模块的代码逻辑分为三个部分，旨在将数据的获取封装，UI，业务逻辑三个部分分离开。
        M即modal:指数据的获取，存储和封装部分
        V即View：指UI页面的构建，刷新和交互部分
        C即Controller: 指控制层，作为V和M之间的桥梁，调度处理View的事件交互结果同步到Modal上。（这个总结OK吗）

MVP模式：（MVP模式 我感觉 多了一个接口层，对于单套实现的业务逻辑 一般不需要这么做 因为这会增加代码的负责）
  把页面模块的代码逻辑分为三个部分，旨在将数据的获取封装、UI展示和业务逻辑三个部分分离开。
  M即Model：指数据的获取，存储和封装部分（与MVC中完全一致，负责数据层的工作）
  V即View：指UI页面的构建，刷新和交互部分，但通常抽象为接口，由Activity/Fragment实现，只负责纯粹的UI操作
  P即Presenter：指呈现器，作为V和M之间的核心协调者，接收View接口转发来的用户事件，从Model获取数据并进行业务处理，再通过View接口回调刷新UI

android： Serializable 和 Parcelable
   这两个两个接口都用于标识数据类的可序列化。
   Serializable  是Java自带的接口采用反射的方式 实现数据的序列化和反序列化
   Parcelable  是android优化后的序列化接口，需要数据类自己实现需要序列化和反序列化的数据，因为没有使用反射，手动代码实现比较复杂，但是性能好。
   (这个总结OK吗)

Dialog 和 PopWindow 的区别：Dialog是自己创建的独立的 PhoneWindow，PopupWindow复用activity的 PhoneWindow.
   PhoneWindow 是Activity的核心成员。

                        2. 接收系统内核的用户触摸事件，向DecorView进行分发
                        3. 状态栏管理和输入法模式管理

APK安装核心过程：
   1. 拷贝APK到：/data/app/包名-随机后缀/base.apk
   2. 签名验证和DEX优化： a. 校验 APK 签名合法性及完整性。b. 执行 dex2oat：将字节码编译为机器码并进行类校验，固化至 .odex/.vdex，后续启动直接运行机器码并跳过验证，从而加速。
   3. 创建数据目录阶段：  创建/data/data/包名/ 应用私有的目录，其他应用无法访问，app 卸载后此目录会被删除。
   4. 资源处理和注册阶段： 资源ID在编译后固定（插件化需要处理资源ID冲突），注册Manifest中声明的四大组件到 PackageManagerService 中，建立组件查找表
   5. 发送安装完成的广播： 通知桌面安装完成 显示正常app启动图标
   （这个总结OK吗）

Android 插件化
    指宿主android App 在运行时，在完全绕过 Android 系统标准安装apk的流程的前提下，通过自定义类加载器（ClassLoading）和 Hook 系统底层
    四大组件的注册和生命周期管理机制，动态加载并运行外部 APK 文件（插件）中的代码与资源的技术方案。
    目前因为 1. Google 官方替代方案 (AAB + Dynamic Delivery)在官方层面支持了“按需下载”和“动态分发”
           2. Hidden API 的“死亡锁”：Android 9.0 到 Android 15，Google 持续收紧了对非 SDK 接口（Hidden API）的访问限制
           完全的插件化机制成本太高，已经不在成为主流技术。但是让宿主apk动态加载外部apk中的的fragment，等不需hook要系统注册管理的api,
           这种低成本的轻量化插件技术，依然是可行的方案。
           即使采用动态加载fragment来实现轻量级的差价化技术，也存在四个问题要解决：
              0. 拷贝到app私有目录前后，两次签名校验保证安全，
              1. 类加载器 (ClassLoader) 的隔离与冲突
              2. 外部的apk要通过自己的资源id 获取自己的布局资源和图片资源，需要构建自己的Resources对象，要构建自己的Resource对象 需要外部apk全路径构建AssetManager对象。
              2. 资源 ID (Resource ID) 的命名空间冲突： 需要通过gradle重新编译器生成资源文件的前缀，以让宿主apk和外部apk的资源文件不会出现相同的资源ID，以实现资源加载校验，不同的Resource对象只能获取指定资源前缀的资源
              3. 外部动态apk如何获通过Resource对象取自己的资源图片，布局
              3. Native 代码 (.so) 的加载路径问题
              4. 宿主与插件的通信协议 (Interface Contract)
           （总结OK吗）

Android 动态加载没有安装的外部的APK机制：
         DexClassLoader类，可以加载指定路径的 .apk,.dex, .jar,zip 文件（PathClassLoader只能加载已经安装的APK(固定了/data/app/路径)）
         DexClassLoader 的核心功能是在进程内建立一套外部 DEX 的类寻址表。它通过 mmap 将磁盘上的 APK/DEX 文件映射到虚拟地址空间，使 ART 虚拟机能够像访问内存一样读取字节码
         DexClassLoader loader = new DexClassLoader(apkPath, optDir, null, getClassLoader());
         Class<?> clazz = loader.loadClass("com.plugin.Test");
         Android开发中，要宿主APP加载执行外部没有安装的APK都是通过 创建 DexClassLoader 对象加载APK文件，然后通过DexClassLoader对象反射调用apk中打包好的Java入口代码。
         如果外部的APK中也有自己的资源图片和资源ID要管理，则需要构建自己的AssetManager和Resource来进行获取，宿主app的Resource对象是无法获取插件资源文件的。

android 性能优化：
   1. UI优化：
           01.ViewStub：懒加载机制，在代码中Inflate的时候 才会创建对象。（对于复杂的条件动态显示布局，优化效果明显）
           02. xml布局优化： 1.include：插入被引用文件的根 View。
           03. merge： 将子引用布局的View列表，直接“注入”到父容器中（引用布局的子元素列表可以没有父元素）。
           04. 复杂布局，尽可能选择ConstraintLayout，减少布局层次优化性能
           05. 谨慎大量在复杂布局中使用LinearLayout 的 layout_weight（权重属性），会测量两次，浪费性能。
           06. 消除过渡绘制，移除布局中，不需要显示的背景属性
           07. 对于复杂的嵌套布局，如果子View会超过父View显示，但是不需要显示超出部分，设置android:clipChildren="true"(默认值)，能帮助系统优化绘图缓存。
   2. 内存优化：
           01. 内存申请是比较耗费资源的，要遵循能不申请就尽量不申请，能少申请尽量少申请。比如对象，可以复用就尽可能复用，而不是创建新的对象，尤其是要注意尽可能避免短时间频繁创建大量的临时对象，这会增加CG压力。
           02. 内存泄漏优化， android开发因为四大组件生命周期的限制，不注意的化，不少代码写法容易出现内存泄漏
           03. 对于对象序列化使用Parcelable(手动实现，最大程度节省内存和CPU)，不要使用Serializable（使用反射浪费内存和性能）。
           04. 对于不同设备不同大小的图片显示区域加载合适分辨率大小的图片，优化内存（现在流行的图片加载缓存框架比如Glide，可以根据图片显示区域大小和图片实际分辨率大小，动态的裁剪压缩图片显示缓存）
