## **一、基础与UI组件**

1. **核心概念**

   * React Native 与 React 的区别是什么？
      React 是用于 Web 的 UI 库，渲染 HTML 元素；React Native 是用于移动端的 UI 框架，
      渲染原生控件，但核心思想、组件化和 JSX 语法是相同的。
   * 什么是 JSX？为什么 RN 里要用 JSX？
      JSX 是 JS 的语法扩展，用 XML/HTML 形式描述 UI，在 React Native 中用它来直观地编写组件结构和布局，
      同时方便 JS 逻辑交互，最终被编译成 React.createElement 渲染到原生控件。
   * RN 中的虚拟 DOM（Virtual DOM）存在吗？工作原理是什么？
       React Native 的 Virtual DOM 是 JS 层的 Fiber 树，用于描述组件结构和状态变化，
       通过 diff 找到最小变更，再通过 Bridge 异步更新原生控件，而不是操作真实 DOM。
   * RN 的渲染机制：JS 线程和原生线程如何协作？
      React Native 的渲染机制是“JS 线程负责逻辑和 Virtual DOM 构建，原生层 UI 线程负责原生控件渲染，
      Bridge 异步传递更新指令和事件”，通过线程分离和异步通信实现高性能和流畅的移动端 UI

1. 热更新相关坑：CodePush 更新失败、版本控制

2. 性能相关坑：FlatList 大量渲染、Animated 阻塞 JS 线程
   一、FlatList 大量渲染导致性能问题
   1. 原因分析
   渲染太多元素 FlatList 会默认渲染 initialNumToRender 个元素，其余按需渲染（虚拟化）。
   如果列表过长或每个 item 组件复杂，会占用大量 JS 和原生线程资源。
   每次 render 都创建新组件/函数：renderItem 中直接定义匿名函数或内联样式，会导致每次列表滚动都触发重新渲染。
   key 不合理： key 不唯一或者随 render 改变 → 导致 element 不复用 → 每次滚动都重新创建组件。
   不合理的 windowSize 或 maxToRenderPerBatch ：windowSize 过大 → JS 线程渲染压力大
   maxToRenderPerBatch 过小 → 列表滚动时卡顿，因为渲染不及时
   复杂子组件：每个 item 内部包含大量子组件或嵌套 FlatList/ScrollView → 渲染量爆炸
