## 22️⃣ Widget 为什么是不可变的？
1. flutter 源码 Widget被 @immutable修饰，被 @immutable 修饰的类及其子类的成员变量 都必须是final的 只能赋值一次，不可再改变。
2. Widget不可变是flutterUI体系的核心规则，为的是让应用层开发不再关注UI组件的中间状态，也就是View复杂的生命周期变化。
3. 让应用层只需要通过配置不可改变的widget来构建UI，实现式UI构建，方便element树复用逻辑的diff 提高性能

flutter 为什么大量 Widget 构造函数是 const？
1. flutter中所有的widget都是不可变的 也就是成员变量都是final的， 那么widget的构造函数就可能使用const修饰（需要widget的所有final修饰的成员变量都在编译期确定才可以）
2. 有些final修饰的的成员变量 需要在运行期间才能确定，这就导致了部分widget的构造函数，不能使用const修饰
3. dart 语言的一个特性是 const修饰的构造函数，在创建对象的时候 也使用const修饰，如果参数都相同，就可以进行内存复用

flutter StatefulWidget 的 state 为什么单独拆？
答案： Widget 是不可变的配置 State 是可变数据 解耦生命周期
      State 的生命周期： State 实例在 Widget 重建时会被 保留（Retain）。只要 Widget 的 runtimeType 和 key 没有变化，
      Flutter 框架就会重用同一个 State 对象，并将新的 Widget 引用挂载到该 State 上。

Flutter 中的 key
 当widget带有 Key后，如果同级兄弟节点的顺序发生变化，Flutter 会进入一个“非顺序匹配”阶段，将带有 Key 的 Element 存入一个 Map<Key, Element> 中，
 通过key 获取匹配（如果数目很大，就可能发生遍历，消耗性能）
flutter 使用widget构建UI树时候，并不是每个widget都需要显示的使用key.不仅不需要，而且通常不建议。
        不建议使用Key的原因：
          1. 性能损耗：Key 的存在会增加框架在 diff 算法中的查找和比较开销
          2. 不使用key 是通过widget在父组件中的index进行迭代索引，性能比Key要好很多
          3. 静态布局没必要：同级多个子widget 是写死的，类型顺序都不会改变
        建议使用Key的情况：
          1.同级节点存在多个相同类型的 StatefulWidget，而且可能发生节点顺序和数目的变化，比如增加或者减少widget
1. ValueKey : 同级兄弟 Widget 的位置可能发生改变的时候，需要使用
2. ObjectKey： 需要对象完全相同的时候 使用
3. UniqueKey： 独一无二的key,强制widget不能复用
3. GlobalKey： 而在于跨树引用，flutter 中widget复用的基础是要有相同的父节点，但是如果widget使用了GlobalKey，即使父节点改变也会保持状态复用


局部刷新 StatefulBuilder & FutureBuilder & StreamBuilder 详解
StatefulBuilder: 提供了setstate用于局部刷新UI，避免为了局部刷新UI创建StatefulWidget类
FutureBuilder： FutureBuilder 接受一个 future对象，在builder中可以获取异步结果的三种状态（成功，失败，获取中），根据不同状态返回对应的UI。
StreamBuilder：StreamBuilder 用于监听一个 Stream，每当Stream的connectionState状态变化（四种）或者有异步结果（成功/失败）的时候，都会触发builder, 返回对应的UI

flutter中BuildContext对象的理解：
 1. 在 Flutter 源码中，BuildContext 实际上是一个接口（Interface），而它的具体实现类就是 Element
 2. 基于 Element 实现了 BuildContext 接口，你可以通过以下 API 进行树操作:
    1. 查找 InheritedWidget (依赖管理):dependOnInheritedWidgetOfExactType<T>() 向上查找最近的类型为 T 的 InheritedWidget，并将当前 Element 注册到其依赖列表中。
    2. 向上遍历 (Ancestry Traversal):  1. findAncestorWidgetOfExactType<T>(): 向上查找最近的指定类型的 Widget
                                     2.findAncestorStateOfType<T>(): 向上查找最近的指定类型的 State 对象
    3. 获取渲染对象信息 (RenderObject)：findRenderObject()：返回当前 Element 关联的 RenderObject，以获取 Widget 在屏幕上的尺寸（size）、全局坐标）等

## **二、GetX 架构与状态管理**
1. **GetX 基础**
   * `Obx`、`GetX`、`GetBuilder` 的区别与使用场景?
      1. GetBuilder: 是非响应式的状态管理器，需要通过关联一个 GetxController 对象，然后显示调用 GetxController 的 update() 函数触发UI刷新。
      2. Obx：响应式管理器，会自动记录闭包中，用到的全部Rx变量，当变量更新的时候 触发UI刷新
      3. GetX:它可以看作是绑定了特定 Controller 的 Obx。响应式管理器，会自动记录闭包中，用到的全部Rx变量，当变量更新的时候 触发UI刷新，但是它提供了绑定和局部注入GetxController的功能。
   * Bindings 的作用和使用方式?
      Bindings: 通过 binding 绑定页面对应的 GetxController,将路由（Route）与 依赖管理（Dependency Injection） 解耦，而且Bindings中可以依赖多个GetxController和支持懒加载机制
                Binding 负责具体的创建逻辑，而泛型（在 GetView<UserController> 或 Get.find 中）负责“寻址”和“匹配”
   * GetX 依赖注入（DI）机制，如何管理 Controller 生命周期。
         GetX中管理 Controller 生命周期的核心 API 是
           1. Get.put：立即实例化并存入内存， 默认随当前路由销毁而销毁。
           2. Get.lazyPut ： 第一次 Get.find 时才执行 Builder 实例化，默认随当前路由销毁而销毁。
           3. Get.find： 获取已经创建的实例，如果没有创建会报错
         Controller 的生命周期钩子：
            1. onInit()  ： widget树还没有初始化完毕，适合进行网络请求初始化、Stream 监听
            2. onReady() ： 在 Widget 渲染到屏幕后的 1 帧调用，适合处理弹窗、引导页等需要 Context 的操作
            3. onClose() ： 当路由销毁，Controller 从内存中移除前调用。必须在此处关闭 StreamController、取消定时器或释放原生资源。
   * 如何实现模块化页面架构（View + Controller + Binding）。

flutter中的三棵树widget element renderObject的关系和作用，以及element renderObject复用规则
   element复用规则： element的复用只有在父节点不变的情况才可能复用，所以无法跨层级复用，
                   如果某个节点的widget卸载了那么这个节点和下面的所有子节点会全部卸载，无法再复用了。
                   Element 复用链是自上而下的： 上层 Element 复用 → 下层 Element 才有机会复用
                                            上层 Element 被替换 → 下层整个子树都跟着重建
